# Ancestry App — `apps/ancestry`

## Purpose

Private family tree web app. Imports data from MyHeritage via GEDCOM export.
Live at `ancestry.dawei7.com`. 421 persons and 144 families imported as of March 2026.

---

## Stack

- **Go 1.24** — HTTP server (`net/http`, Go 1.22 pattern routing)
- **templ v0.3.1001** — HTML templates compiled to Go (run `templ generate` before `go build`)
- **HTMX 2.0.4** — people list live search
- **Tailwind CSS v4.2.2** — standalone CLI, dark zinc/violet theme
- **pgx/v5 v5.7.2** — PostgreSQL driver
- **D3.js v7** — loaded from CDN for the pedigree tree (`cdn.jsdelivr.net/npm/d3@7`)
- **PostgreSQL 17-alpine** — database `dawei_db`
- **Module path**: `github.com/dawei7/dawei7/apps/ancestry`

## Database Schema (`postgres/init/02-ancestry.sql`)

```sql
persons         (id UUID PK, gedcom_id TEXT UNIQUE, first_name CITEXT, last_name CITEXT,
                 maiden_name CITEXT, sex CHAR(1), birth_date TEXT, birth_place TEXT,
                 death_date TEXT, death_place TEXT, notes TEXT, created_at TIMESTAMPTZ)

families        (id UUID PK, gedcom_id TEXT UNIQUE, husband_id UUID→persons,
                 wife_id UUID→persons, marriage_date TEXT, marriage_place TEXT)

family_children (family_id UUID→families, person_id UUID→persons, PK(family_id,person_id))
```

Extensions required (already in `01-init.sql`): `uuid-ossp`, `pg_trgm`, `citext`.

To apply manually to a running container:

```bash
docker exec -i postgres psql -U dawei -d dawei_db < postgres/init/02-ancestry.sql
```

## HTTP Routes

| Method + Path              | Handler                                                                   |
| -------------------------- | ------------------------------------------------------------------------- |
| `GET /`                    | People list shell page                                                    |
| `GET /api/people?q=&page=` | HTMX fragment: paginated/searched people list (50/page)                   |
| `GET /person/{id}`         | Person detail page                                                        |
| `GET /tree/{id}`           | Pedigree tree page shell (D3 loads data client-side)                      |
| `GET /api/tree/{id}?gen=N` | JSON tree for D3. Default: `MaxTreeGens=15`. Returns `NodeJSON` hierarchy |
| `GET /static/tree.js`      | Embedded JS                                                               |
| `GET /static/style.css`    | Embedded CSS                                                              |
| `GET /health`              | `{"status":"ok"}`                                                         |

ID validation: UUID regex check on all `{id}` params before DB query.

## `db/tree.go` — Key Types

```go
type TreeNode struct { Person *Person; Father *TreeNode; Mother *TreeNode }

type NodeJSON struct {
    ID       string      `json:"id"`
    Name     string      `json:"name"`
    Dates    string      `json:"dates,omitempty"`
    Sex      string      `json:"sex,omitempty"`
    Children []*NodeJSON `json:"children,omitempty"` // genealogical parents (ancestors)
}

const MaxTreeGens = 15

func BuildAncestorTree(ctx, pool, personID, depth) (*TreeNode, error)
func MarshalTree(node *TreeNode) ([]byte, error)   // JSON for /api/tree endpoint
func ProbeAncestorDepth(ctx, pool, personID, maxCheck) (int, error)  // cheap probe, unused on page load
```

In the `NodeJSON` hierarchy, "children" = genealogical **parents** (ancestors branch rightward in D3).

## `public/tree.js` — D3 v7 Pedigree Tree

### Concept

- The server returns the **full tree** (all ancestors up to `MaxTreeGens`) in one JSON fetch.
- D3 builds the entire `d3.hierarchy` in memory. Nodes beyond depth 3 (generation 5+) are **collapsed by default** (4 generations visible on load).
- Each node with ancestors has a **+/− circle toggle** on its right edge to expand/collapse that ancestor branch independently.
- No generation slider. User controls depth per-node via the toggle buttons.

### Key constants

```js
((NODE_W = 200), (NODE_H = 56), (GAP_X = 72), (GAP_Y = 14));
DEFAULT_COLLAPSE_DEPTH = 3; // nodes at depth >= 3 start collapsed
```

### Data flow

1. `DOMContentLoaded` → `initSVG()` → `fetch('/api/tree/{rootId}')` → `initHierarchy(data)` → `renderVisible()`
2. `initHierarchy(data)`: builds `d3.hierarchy`, saves `_allChildren` on each node, collapses nodes at `depth >= DEFAULT_COLLAPSE_DEPTH`
3. `renderVisible()`: runs `d3.tree().nodeSize(...)` on the current (partially collapsed) hierarchy, draws links + nodes
4. Toggle click → `toggleNode(id)` → flips `node.children` ↔ `node._allChildren` → `renderVisible()`

### Node rendering details

- Links: bezier from right edge of source to left edge of target (custom path, not `d3.linkHorizontal`)
- Box fill: M=`#1a1f35` (blue), F=`#201a2d` (pink), unknown=`#18181b`
- Top-right `↗` icon: navigates to `/person/{id}`
- Toggle circle: `+` (violet) when collapsed, `−` (grey) when expanded. Only shown on nodes with `_allChildren`.
- Zoom/pan position is **preserved** across toggle re-renders via `savedTransform`

### Selection / filter system

- Click a box (not the toggle, not ↗) → toggles the node into/out of `pending` Set
- `applyTreeSelection()` — computes which nodes are reachable (BFS toward ancestors + leftward chain to root), stores in `activeFilter`, clears `pending`
- `stepBackTree()` — pops the last `filterHistory` entry to undo one apply
- `clearTreeSelection()` — resets everything
- State persisted in `sessionStorage` under key `tree-d3-{rootId}`
- Bottom bar (`#sel-bar`) shows dynamically: Apply / ← Step back / Clear all buttons

### Globals read from page

```js
window.__treeRootId; // injected by tree.templ via templ.Raw(...)
```

## Build Commands

```bash
# Local development
cd apps/ancestry
templ generate          # MUST run first (generates *_templ.go files)
go build ./...

# Docker dev (runs on localhost:3002)
docker compose -f docker-compose.yml -f docker-compose.dev.yml up ancestry --build -d
docker logs ancestry

# Apply schema to running container (only needed once / on new deployments)
docker exec -i postgres psql -U dawei -d dawei_db < postgres/init/02-ancestry.sql

# Import GEDCOM
python scripts/gedcom_import.py C:/Users/david/Downloads/family.ged
# needs: DATABASE_URL env or .env with POSTGRES_* vars
```

> **Always `templ generate` before `go build`.** The `*_templ.go` generated files are what Go compiles — skipping this step builds stale code.
